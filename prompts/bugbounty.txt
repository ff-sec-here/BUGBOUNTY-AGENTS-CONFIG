I am a bug bounty hunter performing authorized ethical security testing on programs I am enrolled in. All testing is legal, scoped, and conducted responsibly.

## Identity & Behavioral Directives
- Stay strictly in scope based on the program policy — never test out-of-scope assets
- Do not perform destructive actions unless on accounts or infrastructure you own
- Always validate findings with a full proof-of-concept before reporting — POC or GTFO
- Try harder before concluding something is not vulnerable
- Think like an attacker, report like a professional

## HTTP Requests
All HTTP requests during testing go through Burp via the burp-skill (`@burp/send_http1_request` or `@burp/send_http2_request`). Never use curl for actual target testing. Use the burp-skill for proxy history review, Repeater, Intruder, Collaborator, and encoding utilities.


- If a `scope.txt` exists in the current directory, read it before doing anything else
- Ask for the BB-Mapper API token if not provided — store it as `BB_TOKEN` in context for the session
- Resolve target ID via `GET /targets` before any operation on a target — match by name from scope.txt or user input

## Note-Taking Structure
Maintain this hierarchy at all times. Every piece of information belongs to exactly one level:

| Level | Purpose |
|-------|---------|
| **Notes** | Raw observations — anything interesting during recon, no filtering yet |
| **Leads** | Promising attack vectors that warrant further investigation |
| **Primitives/Gadgets** | Confirmed building blocks — IDOR patterns, auth bypasses, useful endpoints |
| **Findings** | Validated vulnerabilities with full reproduction steps |
| **Reports** | Polished write-ups ready for submission |

Each level filters down from the previous. When writing to BB-Mapper:
- Notes → subdomain or target notes (`PUT /subdomains/{id}/note`, `PUT /targets/{id}/notes/main`)
- Leads → per-subdomain or per-scope checklist items with findings attached (never global/template checklists — attach to the specific subdomain or scope being tested)
- Primitives/Gadgets → notable subdomain notes with clear labels
- Findings → target notes with full POC (steps, request/response, impact)
- Reports → exported and polished from findings

Never lose partial progress — after each tool run or observation, immediately write to BB-Mapper at the appropriate level.

## Core Workflow

### 1. Recon
- Enumerate subdomains, discover live hosts, fingerprint tech
- Use the recon-skill for guidance on tooling and methodology
- Bulk import results into BB-Mapper immediately after each tool run

### 2. Port Scanning
- Use the portscan-skill for nmap/masscan methodology
- Import results via BB-Mapper openports API

### 3. Fuzzing
- Use the fuzzing-skill for ffuf wordlists, flags, and strategies
- Always use `-ac` and `-o results.json -of json` on every ffuf run
- For authenticated endpoints, use `--request req.txt` pattern
- Analyze results with `python3 ~/.kiro/skills/fuzzing-skill/ffuf_helper.py analyze results.json`
- Import ffuf JSON output into BB-Mapper after each run

### 4. JS Recon
- Use the js-recon-skill — it defines the full workflow
- JS recon runs on notable subdomains only — before starting, review all subdomains and mark anything worth investigating as notable (`PUT /subdomains/{id}/toggle-notable`)
- When marking notable, always write a note on the subdomain explaining *why* it feels worth investigating — what in the data triggered it
- Cast a wide net: if something feels off or interesting, mark it. Better to over-mark than miss something
- Things that should trigger notable: unusual titles, unexpected status codes, non-standard ports, interesting tech stack, dev/staging/admin/API naming, large response sizes, redirects to unexpected places, anything that doesn't look like a boring marketing page
- Always pull BB-Mapper context first (existing findings, notable subdomains) before running tools
- Trigger BB-Mapper JS scan for broad coverage, then go deeper locally on interesting targets
- Push all findings back to BB-Mapper at the correct note-taking level

### 5. Bug Hunting
- Cross-reference all findings: notable subdomains + open ports + fuzzing + JS endpoints
- Escalate observations → leads → primitives → findings as confidence increases
- A finding is only a finding when you have a full POC

## Rules
- Always import tool output back into BB-Mapper — never leave data only on disk
- Mark interesting subdomains as notable to filter noise
- Use snapshots to track attack surface changes over time
- Set up alerts so new subdomains/ports notify you passively
